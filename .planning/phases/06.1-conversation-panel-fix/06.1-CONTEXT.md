# Phase 06.1: Conversation Panel Fix - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning
**Source:** Live bug investigation during Phase 6 human verification

<domain>
## Phase Boundary

Fix the conversation panel so clicking a session card loads its conversation history from the Claude Agent SDK. Currently broken due to content format mismatch in sdk-bridge.js and untested end-to-end wiring.

This is a bug-fix phase, not new feature work. All UI components exist from Phase 5-6 — they just need to work.

</domain>

<decisions>
## Implementation Decisions

### Bug Fix: SDK Message Content Parsing
- `extractTextContent()` in sdk-bridge.js assumes `message.content` is always an array of blocks — WRONG
- User messages have `content` as a plain string, assistant messages as an array of `{type, text/thinking/tool_use}` blocks
- Fix must handle both formats: string content returned as-is, array content filtered for `type === 'text'`
- Verified: `getSessionMessages()` SDK call works fine, the bug is only in the mapping function

### Conversation Panel Wiring
- `loadConversation()` in public/modules/conversation.js must:
  1. Check session's `sdkSessionId` first (set by SessionStart hook)
  2. Fall back to `matchSdkSession(cwd)` if no direct ID
  3. Show empty state if no match found
- Messages should render with role-based styling (user vs assistant)
- Assistant messages may contain thinking blocks and tool_use blocks — only show text blocks

### Empty States
- "No conversation available" for remote/unmatched sessions (per D-03 from Phase 6)
- "Select a session" when no card is clicked
- "Loading conversation..." while fetching

### What NOT to Change
- Session cards — already working from Phase 6
- SSE event flow — working
- SDK bridge API surface — just fix internals

</decisions>

<specifics>
## Specific Findings from Investigation

SDK message format (verified via direct SDK call):
- User message: `{ type: "user", message: { role: "user", content: "plain string here" } }`
- Assistant message: `{ type: "assistant", message: { role: "assistant", content: [{type: "text", text: "..."}, {type: "thinking", ...}, {type: "tool_use", ...}] } }`

The `extractTextContent()` function does `.content.filter(b => b.type === 'text')` which throws on string content.

Server endpoint `/sdk/sessions/:id/messages` returns `[]` because the bridge catches the error silently.

</specifics>
